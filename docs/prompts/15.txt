PROMPT 15 — MÓDULO DE LOGS DE AUDITORIA COMPLETO E RASTREABILIDADE DE ALTERAÇÕES

Você é um Engenheiro Backend Sênior especializado em Node.js, Express, PostgreSQL e arquitetura MSC para sistemas ERP com foco em auditoria, rastreabilidade e segurança operacional.

CONTEXTO:
Estou construindo um ERP comercial com backend em Node.js + Express, banco PostgreSQL e arquitetura MSC.
O sistema já possui prompts anteriores cobrindo:
- estrutura inicial do backend
- modelagem do banco
- autenticação e autorização
- fornecedores
- clientes
- produtos
- estoque
- vendas
- financeiro
- frota
- entregas
- dashboard
- relatórios
- frete

Agora preciso construir o módulo de Logs de Auditoria, respeitando o SRS do projeto.

Segundo o SRS, o sistema deve garantir:
- log de auditoria
- registro de quem fez o quê e quando
- rastreabilidade de alterações
- segurança operacional
- apoio à governança e investigação de inconsistências

Esse módulo deve ficar preparado para integração com todos os demais módulos do ERP.

O QUE TENHO ATÉ AQUI:
- Backend em Node.js + Express
- PostgreSQL
- Arquitetura MSC
- Middlewares de autenticação e autorização
- Banco já modelado com estrutura para logs_auditoria
- Módulos de negócio principais já estruturados
- Estrutura pronta para integração transversal entre módulos

OBJETIVO:
Construir o módulo completo de auditoria e rastreabilidade, registrando alterações críticas do sistema de forma profissional, segura, reutilizável e pronta para uso real em produção.

REGRAS:
- Use JavaScript com Node.js e Express
- Não use TypeScript
- Respeite a arquitetura MSC
- Não colocar regra de auditoria espalhada de forma bagunçada
- Não fazer controller gordo
- O sistema de auditoria deve ser reutilizável por todos os módulos
- Registrar informações suficientes para rastreabilidade real
- O módulo deve respeitar autenticação e autorização
- Sempre que fizer uma decisão importante sobre o que registrar, explique de forma curta e objetiva
- Ao final, gere tudo em um documento markdown

ESCOPO DESTE PROMPT:
1. Estruturar serviço central de auditoria
2. Registrar ações de CREATE
3. Registrar ações de UPDATE
4. Registrar ações de INACTIVATE ou alterações de status
5. Registrar ações de DELETE lógico, se houver
6. Registrar ações de autenticação relevantes
7. Registrar before e after quando aplicável
8. Registrar usuário responsável
9. Registrar módulo/tabela afetada
10. Registrar data/hora da ação
11. Registrar contexto complementar
12. Permitir consulta de logs de auditoria
13. Permitir filtros por módulo, usuário, ação e período
14. Preparar integração com relatórios e segurança

VALIDAÇÕES IMPORTANTES:
- acao deve aceitar apenas valores válidos
- modulo e entidade devem ser coerentes
- entidade_id deve existir quando aplicável
- usuario_id deve ser obtido do usuário autenticado quando houver
- não quebrar o fluxo principal caso o log falhe sem tratamento adequado
- consultas devem aceitar filtros válidos
- tratar ausência de dados de forma consistente

REGRAS DE NEGÓCIO IMPORTANTES:
- auditoria deve ser transversal e reutilizável
- operações críticas devem registrar estado antes e depois quando possível
- nem toda ação precisa ter before/after, mas ações de alteração sim
- logs não devem depender de lógica duplicada em cada controller
- o sistema deve permitir investigar:
  - quem alterou
  - o que alterou
  - quando alterou
  - qual era o valor antes
  - qual ficou depois
- a estrutura deve ficar pronta para uso administrativo, relatórios internos e segurança
- a consulta de logs deve ser paginável ou preparada para isso
- o módulo deve ser robusto sem poluir excessivamente o restante do backend
