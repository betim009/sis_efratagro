PROMPT 05 — CRUD DE CLIENTES

Você é um Engenheiro Backend Sênior especializado em Node.js, Express, PostgreSQL e arquitetura MSC para sistemas ERP.

CONTEXTO:
Estou construindo um ERP comercial com backend em Node.js + Express, banco PostgreSQL e arquitetura MSC.
O sistema já possui:
- estrutura inicial do backend
- modelagem do banco
- autenticação e autorização
- padrão de rotas
- módulo de fornecedores já construído

Agora preciso construir o módulo de Clientes, respeitando o SRS do projeto.

Segundo o SRS, o módulo de Cliente deve permitir:
- cadastrar cliente
- editar cliente
- consultar cliente
- inativar cliente
- controlar status
- armazenar histórico de compras
- armazenar débitos em aberto
- segmentar cliente por tipo:
  - pessoa física
  - pessoa jurídica

Campos esperados no SRS:
- nome/razão social
- CPF/CNPJ
- endereço completo
- telefone
- e-mail
- limite de crédito
- tipo
- status

Status esperados:
- Ativo
- Bloqueado
- Inativo

O QUE TENHO ATÉ AQUI:
- Backend em Node.js + Express
- PostgreSQL
- Arquitetura MSC
- Middlewares de autenticação e autorização
- Banco já modelado
- Controle de permissões
- Estrutura de rotas pronta
- Padrão do CRUD de fornecedores já definido

OBJETIVO:
Construir o CRUD completo de Clientes no backend, de forma profissional, segura, validada e pronta para integração com vendas, financeiro e relatórios.

REGRAS:
- Use JavaScript com Node.js e Express
- Não use TypeScript
- Respeite a arquitetura MSC
- Não colocar regra de negócio diretamente no controller
- Não fazer controller gordo
- Usar validações reais
- Não excluir cliente fisicamente do banco
- Inativação deve ser lógica
- O módulo deve estar pronto para uso real
- Preparar o módulo para integração com histórico de compras e débitos
- Toda rota deve respeitar autenticação e autorização
- Ao final, gere tudo em um documento markdown

ESCOPO DESTE PROMPT:
1. Criar cliente
2. Listar clientes
3. Buscar cliente por id
4. Atualizar cliente
5. Alterar status do cliente
6. Inativar cliente
7. Bloquear cliente
8. Consultar histórico de compras do cliente
9. Consultar débitos em aberto do cliente

CAMPOS ESPERADOS:
- id
- nome_razao_social
- cpf_cnpj
- endereco
- telefone
- email
- limite_credito
- tipo_cliente
- status
- created_at
- updated_at

VALIDAÇÕES IMPORTANTES:
- nome/razão social obrigatório
- CPF/CNPJ obrigatório
- validar duplicidade de CPF/CNPJ
- validar tipo_cliente aceitando apenas:
  - PF
  - PJ
- validar status aceitando apenas:
  - ATIVO
  - BLOQUEADO
  - INATIVO
- validar e-mail, se enviado
- validar limite_credito como número válido
- tratar cliente inexistente corretamente
- impedir atualização inconsistente

REGRAS DE NEGÓCIO IMPORTANTES:
- cliente bloqueado não deve ficar igual semanticamente a cliente inativo
- cliente inativo não deve aparecer como opção normal em fluxos ativos, salvo consulta administrativa
- cliente deve ficar preparado para uso em vendas e financeiro
- histórico de compras e débitos em aberto devem ter endpoint próprio
- se ainda não houver toda a integração implementada, deixar a estrutura pronta de forma coerente com a modelagem existente

ROTAS ESPERADAS NO MÍNIMO:
- POST /clientes
- GET /clientes

- PUT /clientes/:id
- PATCH /clientes/:id/status
- PATCH /clientes/:id/inativar
- GET /clientes/:id/historico-compras
- GET /clientes/:id/debitos-em-aberto

ARQUIVOS ESPERADOS NO MÍNIMO:

- src/routes/clienteRoutes.js
- validações e helpers necessários

PERMISSÕES ESPERADAS COMO EXEMPLO:

- clientes.inactivate
- clientes.block
- clientes.financial.read

- Não apagar cliente com DELETE físico
- Não deixar CPF/CNPJ duplicado
- Não tratar bloqueado e inativo como a mesma coisa
- Não quebrar o padrão do backend já criado
- O resultado precisa ficar pronto para crescimento real dentro do ERP
