PROMPT 16 — MÓDULO DE NOTIFICAÇÕES INTERNAS E BASE PARA ENVIO POR E-MAIL

Você é um Engenheiro Backend Sênior especializado em Node.js, Express, PostgreSQL e arquitetura MSC para sistemas ERP com foco em notificações operacionais, alertas internos e integração futura com envio por e-mail.

CONTEXTO:
Estou construindo um ERP comercial com backend em Node.js + Express, banco PostgreSQL e arquitetura MSC.
O sistema já possui prompts anteriores cobrindo:
- estrutura inicial do backend
- modelagem do banco
- autenticação e autorização
- fornecedores
- clientes
- produtos
- estoque
- vendas
- financeiro
- frota
- entregas
- dashboard
- relatórios
- frete
- auditoria

Agora preciso construir o módulo de Notificações, respeitando o SRS do projeto.

Segundo o SRS, o sistema deve possuir alertas e notificações para eventos como:
- estoque abaixo do mínimo
- venda futura próxima do vencimento
- duplicata vencida
- manutenção preventiva pendente
- entrega não concluída
- atualização relevante de status de entrega

Também existe a possibilidade futura de:
- envio por e-mail
- integração com dashboard
- central de notificações no frontend

Esse módulo deve ficar preparado para integração transversal com todos os módulos do ERP.

O QUE TENHO ATÉ AQUI:
- Backend em Node.js + Express
- PostgreSQL
- Arquitetura MSC
- Middlewares de autenticação e autorização
- Banco já modelado
- Módulos principais do ERP já estruturados
- Dashboard e relatórios já previstos
- Auditoria já prevista
- Estrutura pronta para integração transversal entre módulos

OBJETIVO:
Construir o módulo completo de notificações internas, com leitura, listagem, marcação como lida, geração por gatilho e base preparada para expansão futura com e-mail.

REGRAS:
- Use JavaScript com Node.js e Express
- Não use TypeScript
- Respeite a arquitetura MSC
- Não colocar lógica de notificação espalhada de forma bagunçada
- Não fazer controller gordo
- O sistema de notificações deve ser reutilizável por todos os módulos
- O módulo deve respeitar autenticação e autorização
- As notificações devem ter estrutura consistente para o frontend consumir
- Sempre que fizer decisão importante sobre gatilhos, explique de forma curta e objetiva
- Ao final, gere tudo em um documento markdown

- por exemplo:
  - estoque baixo → estoquista e gerente
  - duplicata vencida → financeiro e gerente
  - entrega não concluída → vendedor responsável
  - manutenção pendente → responsável da frota
- a estrutura deve servir para consumo no frontend, incluindo badge de não lidas
- deve existir contagem de notificações pendentes
- a solução deve ficar preparada para extensão futura com e-mail, sem acoplar agora o envio real
- as notificações devem ser facilmente integráveis ao dashboard
- evitar duplicação desnecessária de lógica em cada módulo
