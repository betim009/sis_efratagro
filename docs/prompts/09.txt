PROMPT 09 — MÓDULO FINANCEIRO DE DUPLICATAS E PAGAMENTOS

Você é um Engenheiro Backend Sênior especializado em Node.js, Express, PostgreSQL e arquitetura MSC para sistemas ERP com foco em financeiro, contas a receber e rastreabilidade de pagamentos.

CONTEXTO:
Estou construindo um ERP comercial com backend em Node.js + Express, banco PostgreSQL e arquitetura MSC.
O sistema já possui:
- estrutura inicial do backend
- modelagem do banco
- autenticação e autorização
- módulos de fornecedores, clientes, produtos, estoque e vendas
- padrão profissional de rotas, services, models e controllers

Agora preciso construir o módulo Financeiro, com foco em duplicatas e pagamentos, respeitando o SRS do projeto.

Segundo o SRS, o módulo Financeiro deve permitir:
- gerar duplicata automaticamente a partir de uma venda a prazo
- registrar campos como número, valor, vencimento, cliente, venda de origem e status
- manter status da duplicata:
  - EM_ABERTO
  - PAGO_PARCIALMENTE
  - PAGO
  - VENCIDO
- registrar pagamentos com data e forma de pagamento
- listar duplicatas por cliente, período e status
- alertar duplicatas vencendo nos próximos X dias

Esse módulo deve ficar preparado para integração total com vendas, clientes, relatórios e dashboard.

O QUE TENHO ATÉ AQUI:
- Backend em Node.js + Express
- PostgreSQL
- Arquitetura MSC
- Middlewares de autenticação e autorização
- Banco já modelado
- Controle de permissões
- Módulo de vendas já preparado para vendas à vista, a prazo e parceladas
- Estrutura pronta para integração entre módulos

OBJETIVO:
Construir o módulo completo de duplicatas e pagamentos, de forma profissional, segura, auditável, consistente financeiramente e pronta para uso real no ERP.

REGRAS:
- Use JavaScript com Node.js e Express
- Não use TypeScript
- Respeite a arquitetura MSC
- Não colocar regra de negócio diretamente no controller
- Não fazer controller gordo
- Usar validações reais
- Toda duplicata deve manter consistência com a venda de origem
- Todo pagamento deve atualizar corretamente o saldo e o status da duplicata
- Não permitir pagamento inconsistente
- Toda rota deve respeitar autenticação e autorização
- O módulo deve ficar preparado para integração com dashboard, relatórios e vendas
- Ao final, gere tudo em um documento markdown

VALIDAÇÕES IMPORTANTES:
- venda_id deve existir para geração vinculada
- cliente_id deve existir
- valor_original deve ser número válido maior que zero
- vencimento obrigatório
- status deve aceitar apenas valores válidos
- pagamento deve ter valor maior que zero
- não permitir pagamento acima do saldo em aberto, a menos que você implemente e justifique política de crédito ou troco interno
- forma_pagamento deve aceitar apenas valores válidos
- duplicata deve existir
- não permitir pagamento em duplicata cancelada, se você decidir suportar cancelamento e justificar
- tratar duplicata inexistente corretamente
- tratar venda inexistente corretamente

REGRAS DE NEGÓCIO IMPORTANTES:
- duplicata pode nascer automaticamente a partir de uma venda a prazo
- venda parcelada deve permitir múltiplas duplicatas
- valor_em_aberto deve ser recalculado corretamente
- status deve mudar automaticamente conforme os pagamentos:
  - sem pagamento e dentro do prazo = EM_ABERTO
  - sem quitação total e com parte paga = PAGO_PARCIALMENTE
  - quitação total = PAGO
  - vencida e não quitada = VENCIDO
- a soma dos pagamentos não deve quebrar a consistência do financeiro
- toda operação deve manter rastreabilidade de usuário e data
- estrutura deve ficar pronta para alertas de vencimento
- estrutura deve ficar pronta para uso em dashboard e relatórios
